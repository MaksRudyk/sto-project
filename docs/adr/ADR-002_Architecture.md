#
ADR-002: Вибір монолітної архітектури
 
##
Статус
Accepted
 
##
Контекст
Проєкт розробляється невеликою командою (3 особи) з метою швидкого виходу на ринок (MVP) за 4 місяці. Необхідна висока швидкість обміну даними між модулями (авторизація, статті, коментарі).
 
##
Рішення
Використовувати монолітну архітектуру з чітким поділом на шари (Presentation, Business Logic, Data Access).
 
##
Обґрунтування
1. Швидкість розробки: Немає потреби налаштовувати мережеву взаємодію між мікросервісами.
2. Простота розгортання: Весь додаток "AutoSport Hub" деплоїться як одна одиниця.
3. Цілісність транзакцій: Простіше реалізувати бізнес-логіку створення статті та сповіщення в одній транзакції.
 
##
Альтернативи
- Мікросервіси: Відхилено через надмірну складність для поточного масштабу та команди.
- Serverless: Відхилено через проблему холодного старту, що критично для інтерактивного форуму.
 
##
Наслідки
###
Позитивні
- Мінімальні витрати на інфраструктуру на старті.
- Простіший моніторинг помилок.
###
Негативні
- При значному зростанні аудиторії може знадобитися складний рефакторинг або поділ на сервіси.
